# Industry: Tech / SWE / ML / Data

Applies to: software engineer, ML engineer, data scientist, data engineer, infra/platform, SRE, DevOps, security, research engineer at a tech company, PM (technical).

## Length
- **≤ 7 years experience:** 1 page, hard cap.
- **Senior / staff / principal:** 1.5–2 pages. Never exceed 2.
- Big Tech (Google, Meta, Amazon, etc.) reviewers often spend <60 seconds — density beats completeness.

## Order of sections
1. One-line summary (skip if fresh grad; use an objective only for career-switchers)
2. Experience (reverse chronological)
3. Projects (only if they strengthen the pitch — e.g., for career-switchers or research→industry)
4. Education
5. Skills
6. Publications / patents (only if directly relevant; otherwise a one-liner)
7. Awards (only top 2–3)

For research-heavy roles (research scientist at a lab), publications move above Skills.

## Bullet style
- Start with an **action verb in past tense** (present tense only for current role).
- Structure: **verb + what + impact / metric**.
- Prefer **quantified outcomes**: latency reductions, cost savings, QPS, users, revenue, model accuracy.
- Prefer concrete tools: "PyTorch", "CUDA", "Kafka", "gRPC", "Terraform" — ATS keyword hits matter.
- Avoid: "responsible for", "helped with", "worked on a team that…".

Good: *"Reduced p99 inference latency 340ms → 85ms by fusing CUDA kernels; shipped to 1.2B daily requests."*
Bad: *"Worked on improving inference performance for the ML team."*

## Keywords
ATS filters at tech firms (Greenhouse, Lever, Workday) lean heavily on exact keyword match. Promote the posting's exact phrasing when it's accurate — e.g., if the posting says "LLM post-training" and your master says "RLHF pipelines", use **both**.

Technologies worth name-dropping by tier:
- **Cloud**: AWS / GCP / Azure (name the services used — not just "cloud")
- **Languages**: Python, Go, Rust, C++, Java, TypeScript (match role level)
- **ML**: PyTorch, JAX, TensorFlow, Hugging Face, Ray, vLLM, Triton
- **Infra**: Kubernetes, Docker, Terraform, Airflow, Spark, Kafka, Flink
- **Data**: BigQuery, Snowflake, Redshift, ClickHouse, Postgres, DuckDB

## Publications & patents
- **Research role**: full list (selective co-authors), grouped by venue tier (top-tier first).
- **Applied/industry role**: 2–3 most relevant first-author papers, or a one-line summary in the header.
- **Pure SWE role**: drop the section; mention in summary only if the user has a standout (NeurIPS first-author, etc.).

## GitHub / portfolio
- Include a GitHub link if the profile shows real, relevant code. Don't link a graveyard.
- Personal site / blog is a plus if technical; remove if it's a dev log of half-finished projects.

## Photo / DOB / marital status
**Never** on a US-targeted tech resume. For DE/CH/AT tech postings, photo is traditional but optional — follow the user's existing master-resume convention.

## Red flags to avoid
- Buzzword soup ("synergistic cross-functional stakeholder alignment")
- Responsibility bullets with zero outcome
- Skill lists that contradict the experience ("expert in Rust" with no Rust bullet)
- Stale tech that dates the candidate (jQuery, Flash, Perl unless relevant to a specific employer)
- Mentioning tools only used "in a course" without lab or work experience — drop them.
